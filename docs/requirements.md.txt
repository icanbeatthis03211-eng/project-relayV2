# 📘 [기획 진술서] Project Relay: 프로젝트형 부트캠프 수강생을 위한 피드백 성장 관리 서비스

## 1. 서비스 개요

* **서비스명:** Project Relay

* **서비스 개요:**
  Project Relay는 프로젝트형 부트캠프 수강생이 받은 피드백을 저장하고, 반복적으로 등장하는 피드백을 찾아 다음 프로젝트 체크리스트로 바꿔주는 성장 관리 웹 서비스이다.

  사용자는 튜터 피드백, 팀원 피드백, 발표 피드백, 회고 내용을 간단히 저장할 수 있다. 저장된 피드백은 역량 태그별로 정리되고, 같은 태그가 반복될 경우 “내 반복 피드백”으로 표시된다.

  이 서비스의 핵심 가치는 공유 데이터가 충분히 쌓이기 전에도 사용자가 혼자서 자신의 피드백 패턴을 확인할 수 있다는 점이다. 사용자는 자신이 자주 지적받는 부분을 확인하고, 다음 프로젝트 전에 무엇을 점검해야 하는지 체크리스트로 받아볼 수 있다.

  또한 공유 가능한 피드백은 익명 카드로 변환해 다른 수강생들과 공유할 수 있다. 공유된 피드백이 쌓이면 프로젝트 유형별로 자주 반복되는 피드백을 확인할 수 있는 학습 라이브러리로 확장된다.

---

## 2. 핵심 기능 및 화면 요구사항

### 1) 메인 화면 / 대시보드

메인 화면에서는 사용자의 피드백 성장 현황을 한눈에 보여준다.

**필수 요소:**

* 서비스명과 한 줄 소개
* [피드백 저장하기] 버튼
* 저장된 피드백 수
* 반복 피드백 태그 수
* 생성된 체크리스트 항목 수
* 공유 가능한 피드백 카드 수
* 서비스 사용 흐름 안내

**화면에 보여줄 흐름 예시:**

피드백 저장
→ 반복 피드백 감지
→ 다음 프로젝트 체크리스트 생성
→ 공유 가능한 피드백 카드 생성

이 화면은 Project Relay가 단순한 피드백 메모장이 아니라, 피드백을 성장 행동으로 바꿔주는 서비스임을 보여줘야 한다.

---

### 2) 피드백 저장 화면

사용자가 자신이 받은 피드백을 저장하는 화면이다.

**입력 항목:**

* 프로젝트 유형
  예: 역기획 프로젝트, 데이터 드리븐 프로젝트, MVP 프로젝트, 최종 프로젝트

* 피드백 출처
  예: 튜터, 팀원, 발표 피드백, 자기 회고

* 원본 피드백 내용
  사용자가 받은 피드백을 그대로 입력하거나 붙여넣는다.

* 역량 태그
  예: 문제 정의, 타깃 설정, 리서치 근거, KPI, 우선순위, 솔루션 논리, 데이터 해석, 사용자 관점, 발표 흐름, 장표 표현

* 공유 가능 여부
  이 피드백을 익명 공유 카드로 만들 수 있는지 선택한다.

**핵심 UX 원칙:**

사용자가 긴 회고문을 작성해야 한다고 느끼면 안 된다.
피드백 입력은 짧고 가볍게 끝나야 한다.

---

### 3) 내 반복 피드백 화면

저장된 피드백을 바탕으로 사용자가 반복해서 받은 피드백을 보여주는 화면이다.

**필수 요소:**

* 역량 태그별 피드백 개수
* 같은 태그가 2회 이상 등장하면 반복 피드백으로 표시
* 가장 많이 반복된 피드백 영역 강조
* 각 반복 태그와 관련된 최근 피드백 로그 표시
* 간단한 해석 문구 제공

**예시 출력:**

* 문제 정의 — 3회 반복
  “문제 범위를 좁히는 것과 관련된 피드백이 반복되고 있어요.”

* 타깃 설정 — 2회 반복
  “타깃 사용자를 더 구체화할 필요가 있어 보여요.”

* KPI — 1회
  “아직 반복 피드백은 아니지만, 다음 프로젝트에서 함께 점검하면 좋아요.”

**MVP 기준:**

초기 버전에서는 같은 태그가 2회 이상 등장하면 반복 피드백으로 판단한다.

---

### 4) 다음 프로젝트 체크리스트 화면

반복 피드백을 다음 프로젝트 전에 확인할 체크리스트로 변환하는 화면이다.

**필수 요소:**

* 반복 피드백 태그 기반 체크리스트
* 체크박스 인터랙션
* 각 체크리스트 항목이 왜 필요한지에 대한 짧은 설명
* 새 프로젝트 시작 시 체크리스트 초기화 기능

**체크리스트 예시:**

* 문제 정의가 반복된 경우
  “문제를 한 문장으로 명확하게 좁혔는가?”

* 타깃 설정이 반복된 경우
  “타깃 사용자의 상황, 불편, 행동을 구체적으로 설명할 수 있는가?”

* KPI가 반복된 경우
  “설정한 KPI가 해결안과 직접적으로 연결되어 있는가?”

* 리서치 근거가 반복된 경우
  “주요 주장마다 최소 1개 이상의 근거 자료가 있는가?”

* 발표 흐름이 반복된 경우
  “발표 흐름이 Problem → Insight → Solution 순서로 이어지는가?”

이 화면은 사용자가 과거 피드백을 다음 프로젝트 준비에 바로 활용할 수 있게 만들어야 한다.

---

### 5) 공유 가능한 피드백 카드 화면

사용자가 선택한 피드백을 익명 공유 카드로 변환하는 화면이다.

**필수 요소:**

* 프로젝트 유형
* 일반화된 피드백 요약
* 역량 태그
* 다음 프로젝트에서 조심할 점
* [카드 복사하기] 버튼
* [공유 라이브러리에 추가] 버튼

**중요한 공유 원칙:**

원본 피드백을 그대로 공유하지 않는다.
개인 정보나 민감한 내용을 제거하고, 다른 수강생도 참고할 수 있는 학습 포인트만 남긴다.

**공유 가능한 정보:**

* 프로젝트 유형
* 일반화된 피드백
* 역량 태그
* 다음 액션 아이템

**숨겨야 하는 정보:**

* 이름
* 팀명
* 튜터명
* 구체적인 서비스명
* 개인 평가처럼 느껴지는 문장
* 민감한 프로젝트 정보

공유의 목적은 개인 피드백을 공개하는 것이 아니라, 반복될 수 있는 시행착오를 학습 자료로 바꾸는 것이다.

---

### 6) 공유 피드백 라이브러리 화면

익명 공유된 피드백 카드를 모아보는 화면이다.

**필수 요소:**

* 공유 피드백 카드 목록
* 프로젝트 유형별 필터
* 역량 태그별 필터
* 공유된 피드백 중 반복되는 패턴 표시
* 공유 데이터가 부족할 때의 빈 상태 메시지

**핵심 UX 원칙:**

공유 라이브러리는 커뮤니티 게시판처럼 보이면 안 된다.
댓글, 좋아요, 팔로우, DM 기능은 MVP에 포함하지 않는다.

이 화면은 소셜 기능이 아니라, 다른 수강생들의 시행착오를 참고할 수 있는 학습 데이터베이스처럼 보여야 한다.

---

## 3. 백엔드 연동 로직

### 1) 피드백 데이터 저장

사용자가 피드백을 저장하면 `feedbacks` 테이블에 데이터가 저장된다.

**예시 테이블: `feedbacks`**

* `id`
* `user_id`
* `project_type`
* `feedback_source`
* `original_feedback`
* `summary`
* `tag`
* `action_item`
* `is_shareable`
* `created_at`

저장된 피드백은 이후 내 반복 피드백 분석의 기준 데이터가 된다.

---

### 2) 반복 피드백 감지

사용자가 내 반복 피드백 화면에 진입하면, 해당 사용자가 저장한 피드백을 불러와 역량 태그별로 그룹화한다.

**MVP 기본 로직:**

* 사용자의 피드백을 태그별로 집계한다.
* 같은 태그가 2회 이상 등장하면 반복 피드백으로 판단한다.
* 반복 횟수가 많은 순서대로 정렬한다.
* 가장 반복된 태그를 상단에 보여준다.

**예시:**

* 문제 정의: 3회 → 반복 피드백
* 타깃 설정: 2회 → 반복 피드백
* KPI: 1회 → 관찰 필요

이 구조를 통해 공유 데이터가 충분히 쌓이기 전에도 사용자는 자신의 피드백 패턴을 확인할 수 있다.

---

### 3) 체크리스트 생성

반복 피드백이 감지되면, 태그별 체크리스트 매핑을 바탕으로 다음 프로젝트 체크리스트를 생성한다.

**예시 매핑:**

* 문제 정의 → “문제를 한 문장으로 명확하게 좁혔는가?”
* 타깃 설정 → “타깃 사용자의 상황과 문제를 구체적으로 설명할 수 있는가?”
* KPI → “KPI가 해결안과 직접 연결되어 있는가?”
* 리서치 근거 → “주요 주장마다 근거 자료가 있는가?”
* 발표 흐름 → “발표가 Problem → Insight → Solution 흐름으로 이어지는가?”

체크리스트는 별도 `checklists` 테이블에 저장하거나, 저장된 피드백을 기반으로 동적으로 생성할 수 있다.

---

### 4) 공유 카드 생성

사용자가 피드백을 공유 가능으로 설정하면, 해당 피드백을 바탕으로 익명 공유 카드가 생성된다.

**예시 테이블: `shared_cards`**

* `id`
* `feedback_id`
* `project_type`
* `generalized_feedback`
* `tag`
* `action_item`
* `created_at`

원본 피드백은 개인 공간에만 남고, 공유 라이브러리에는 일반화된 학습 포인트만 저장된다.

---

### 5) 공유 라이브러리 데이터 로딩

사용자가 공유 피드백 라이브러리에 들어가면 `shared_cards` 테이블에서 익명 카드 데이터를 불러온다.

**라이브러리 표시 로직:**

* 전체 공유 카드 불러오기
* 프로젝트 유형별 필터 적용
* 역량 태그별 필터 적용
* 공유 카드의 태그 빈도 집계
* 충분한 데이터가 쌓이면 자주 공유된 피드백 패턴 표시

이 구조는 공유 기능을 포함하되, 서비스의 핵심 가치를 공유 데이터에만 의존하지 않게 만든다.

---

## 4. MVP 범위

MVP는 개인 피드백 가치에 먼저 집중한다.

**MVP에 반드시 포함할 기능:**

1. 피드백 저장
2. 역량 태그별 피드백 정리
3. 내 피드백 안에서 반복 피드백 감지
4. 다음 프로젝트 체크리스트 생성
5. 공유 가능한 피드백 카드 생성
6. 공유 피드백 라이브러리 확인

**MVP에서 제외할 기능:**

* AI 자동 태그 추천
* Notion 연동
* 튜터용 대시보드
* 팀별 비교
* 댓글, 좋아요 기능
* 실시간 채팅
* 고급 분석 리포트

---

## 5. 핵심 가치 제안

Project Relay는 흩어진 피드백을 다음 프로젝트에 활용할 수 있는 성장 데이터로 바꿔준다.

사용자는 피드백을 단순히 저장하는 데서 끝나지 않고, 자신이 반복해서 받는 피드백이 무엇인지 확인할 수 있다. 그리고 그 반복 피드백을 바탕으로 다음 프로젝트 전에 확인해야 할 체크리스트를 얻을 수 있다.

또한 공유 가능한 피드백은 익명 카드로 전환되어 다른 수강생에게도 도움이 되는 학습 자료가 된다.

즉, Project Relay는 혼자 써도 자신의 반복 피드백을 확인할 수 있고, 공유하면 부트캠프 전체의 학습 데이터로 확장되는 서비스이다.

---

## 6. 한 줄 컨셉

Project Relay는 부트캠프 수강생이 받은 피드백을 저장하고, 반복되는 피드백을 찾아 다음 프로젝트 체크리스트로 바꿔주는 피드백 성장 관리 서비스이다.
